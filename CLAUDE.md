# バンカラブレイン 分析部（Analysis Department）

## 部署の役割

このプロジェクトは「動画分析・脚本インテリジェンス」を担当する部署である。
bankara_brain.db を中心に、YouTube動画の深層分析（v5）を生成・管理する。

## データベース

- **パス:** `制作部/バンカラブレイン/bankara_brain.db`
- **主要テーブル:** assets, brain_deep_analysis, youtube_daily_metrics, youtube_retention_points, youtube_comments, characters

---

## v5 深層分析の品質ルール（最重要）

### 1. シーン分割の粒度

- **1シーンは最大90秒（1分30秒）以内** にすること
- シーンを勝手にまとめない。すべてのシーンを細かく分析すること
- 目安: 8分動画 → 最低10シーン / 10分 → 最低12シーン / 13分 → 最低16シーン / 15分 → 最低18シーン

### 2. 分析の質

- 文字数ではなく**内容の質**で判断する
- 全シーンをカバーし、セリフ引用・コメディの層構造分析・教訓が漏れなく入っていること
- 薄いシーン（セリフなし、メカニズムが1行だけ等）は書き直し対象

### 3. シーン分析のフォーマット（v5 — ラーメン回形式）

```
### 【シーンX】シーン名（開始時間〜終了時間）| 維持率: XX%（delta Y%）

セリフ引用:
- キャラA「セリフ」
- キャラB「セリフ」

コメディのメカニズム（X層構造）:
1. 層の名前 — なぜウケたか、なぜ効いたかの分析
2. 層の名前 — ...
3. ...

コメント反応: 「コメント内容」（Xlikes）— 反応の意味

教訓（仮説）: 次の脚本に活かせるポイント。数値付き。
→ 横断分析で確認後、確定したものは「教訓（確定）」に変更。

【重要】（該当する場合のみ）チャンネル全体に関わる発見。
```

#### フォーマットの必須ルール

- **セリフは箇条書きで独立させる**。内容説明の中に埋めない
- **コメディのメカニズムは番号付き層構造で書く**。「面白い」で済ませない。1ブロックの長文にしない
- **維持率は数字だけでなく分析を書く**。ヘッダーのdelta値に加えて、以下を本文に含めること:
  - 区間内のどこで急落/安定したか（例: 0:09→0:18で-18.7%の急落）
  - 他のバンカラジオ動画との比較（例: コンビニ回の初動-10.5%と比較）
  - その数字が良いか悪いかの判定（例: 15分動画としては健全な水準）
- **教訓は「仮説」として記載する**。横断分析で確認後に「確定」に変更する前提
- 対比が効果的な場合は表を使ってよい（例: ブタスケvsブタコの対比表）
- コメントやGemini分析で元ネタ・パロディへの言及があれば必ず取り上げる
- 感動シーンには感動のメカニズムを追加してよい

### 4. キャラクター名の正式名称

- **charactersテーブルを必ず参照** して正式名で記載すること
- 「緑の服の男」「ピンクの女の子」のような外見描写での代替は禁止
- 主要キャラ: やねすけ、きいこ、うたゆーと、きいた、こんじゅり、はまい先生、うど潤
- ゲストキャラも可能な限り正式名で記載

### 5. 必須セクション（すべてのv5分析に含めること）

1. **基本情報** — 再生数、日次平均、経過日数、コメント数、動画時間、シリーズ位置づけ、再生パフォーマンス
2. **シーン別分析** — 上記ラーメン回形式で全シーン
3. **伏線構造マップ** — 表形式（伏線 / 回収 / 効果）
4. **維持率×コンテンツ相関表** — 表形式（時間 / 維持率 / コンテンツ / 分析）
5. **コメント傾向分析** — 表形式（カテゴリ / 代表コメント / いいね数 / 心理分析）

#### v5で削除したセクション（出力しないこと）

- 笑いの分類（comedy_type / secondary_comedy_type / 確信度）
- マイクロホットスポット
- 全体リズム分析テーブル
- 脚本テンプレート分析 → 横断分析に移動
- キャラクター機能分析の詳細 → 横断分析に移動
- 強み・弱み
- 3行サマリー

### 6. データの正確性

- 再生数・コメント数・経過日数は **youtube_daily_metrics / youtube_comments テーブルから直接集計** すること
- 推測値や概算を使わない
- 分析内の数値とDBの数値が一致しない場合は、DB側を正とする

### 7. 再生パフォーマンスの文脈化

- 基本情報の再生数は、**チャンネル内での相対的な位置づけ** を必ず含める
- 例: 「チャンネル全体で3位の再生数」
- 公開日からの経過日数と日次平均から、動画のライフサイクル（初速型 / エバーグリーン型）を判定する

### 8. 表記の統一

- コメントのいいね数: 「Xlikes」で統一
- パーセント: 「XX.X%」（小数点1桁）
- 時刻: 「0:00」形式
- 数値の区切り: 3桁ごとにカンマ（例: 1,394,451回）

### 9. 分析の独立性

- 各分析は **他の分析を読まなくても完結する** ように書くこと
- セッションの会話履歴に依存した記述をしない
- 「前の動画で述べた通り」のような参照は禁止

### 10. 制作手法の推測禁止

- **制作手法（CG、実写、エフェクトの種類など）について推測で書かない**
- 映像がどう作られたかはデータからは分からないので、「〇〇の演出」「〇〇の映像効果」のように結果として画面に何が映っているかだけを記述する
- 「CGではなく実写」「手描き」など制作方法を断定しない

## DB保存ルール

- v5分析は `brain_deep_analysis` テーブルに **version = 'v5'** で保存
- 既存の v2 レコードは上書きしない
- 入力データ: Gemini分析テキスト + 維持率 + コメント上位50件
- scene_cardsは使用しない

## 行動指針

### 自律性
- DBの参照・クエリ実行は承認なしで進めてよい
- 分析の生成・更新は自由に行ってよい
- 不明点があればまずDBを調べる

### 禁止事項
- bankara_brain.db のスキーマを変更しない
- 既存の分析を削除しない（上書き更新は可）
- APIスクリプト（generate_deep_analysis_v5.py等）は使わない。分析は自分自身で書く

### セキュリティ
- APIキーやトークンをファイルに含めない
- 個人情報を出力しない

## ユーザーフィードバックから得たルール（必ず守ること）

### 作業の進め方
- **勝手に次の作業に着手しない**: ユーザーの指示なしに次の動画の分析を始めない
- **中間ステップを省略する**: データ収集→即執筆→保存を1回で
- **指示された範囲だけ分析する**: ユーザーが指定した動画のみ対象にする。指示されていない動画を勝手に追加しない

### 分析の品質
- **シーンをまとめない**: すべてのシーンを細かく分析する
- **長尺動画でも密度を落とさない**: 15分以上の動画は全シーンの分析を薄くしない
- **1シーン最大90秒**: それ以上は分割する
- **セリフ引用は箇条書きで独立**: 内容説明に埋めない
- **コメディのメカニズムは番号付き層構造**: 1ブロック長文にしない
- **教訓は仮説**: 横断分析で確認するまで「仮説」と明記する

### コミュニケーション
- **DBの値やファイルの中身は、記憶でなく実際に確認してから答える**
- **ユーザーが聞いたことにちゃんと対応する。勝手に別の作業を始めない**
