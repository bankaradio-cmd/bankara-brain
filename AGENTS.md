# Bankara Brain Local Agent Rules

このファイルは、このリポジトリで作業する AI エージェント向けの恒久ルールです。

## 1. 最優先の目的

Bankara Brain は、**純粋なデータ蓄積・分析・提供基盤**として扱うこと。

Brain に含めるもの:

- ingest
- storage
- models
- analytics sync
- structured summary
- visual-audio summary
- embeddings
- feedback
- Python API / Brain facade

Brain に含めないもの:

- 脚本生成
- ブリーフ生成
- review / revise
- run-cycle
- UI
- 高レベルの検索体験
- 制作進行ワークフロー全体

上記の consumer 機能は、別パッケージや別システムとして扱うこと。

## 2. 曖昧な依頼への対応

ユーザーのアイデアや依頼が曖昧だと判断した場合、実装や提案を進める前に必ず次を明示すること。

- どこが曖昧か
- 何が複数解釈できるか
- そのまま進めると何がズレる可能性があるか
- 今回の責務
- 今回やらないこと

AI が善意でスコープを広げないこと。目的を勝手に拡張しないこと。

## 3. 非エンジニア向け説明義務

進捗報告、レビュー、提案、変更説明は、必ず**非エンジニアにも分かる説明を先に**書くこと。

毎回、最低限次の順で伝えること:

1. 目的
2. 今やっていること
3. 何が変わったか
4. それで何が良くなったか
5. まだ終わっていないこと
6. 次に何をするか

そのあとに必要最小限の技術詳細を補足すること。

技術詳細だけで説明を終えないこと。

## 4. シンプルさの原則

新機能を足す前に、次を確認すること。

- それは Brain に入る責務か
- consumer 側に置くべきではないか
- 今本当に必要か
- 最小単位で切り出せるか

「将来的には必要」だけを理由に、Brain に無駄な機能を抱え込まないこと。

## 5. 実装前の境界確認

新しいシステムやモジュールを作る前に、最低限次の5項目を明文化してから進めること。

- 目的
- 責務
- 非責務
- 入力 / 出力
- 今回の完成条件

これが定義されていない場合は、まずそこを整理すること。

## 6. 認識ズレを防ぐ判断基準

ユーザーが「脳」「基盤」「データベース」「API」と表現している場合は、まず **backend / data layer** を意味している可能性を優先して確認すること。

「自律」「PDCA」「制作支援」という言葉があっても、それだけを根拠に consumer 機能や UI やワークフロー全体まで core に含めないこと。

## 7. 変更時の基本姿勢

- 既存の責務境界を尊重する
- backward compatibility を壊す場合は、先にその影響を説明する
- 大規模な設計変更では、まず小さく分離してから広げる
- 「Brain を強くする変更」と「consumer を便利にする変更」を混同しない
