# 「動画理解レイヤー」構築計画

## 現状の問題

今のバンカラブレインは「データを保存しているだけ」。

- VAS分析（brain_summary_v1）は、セリフの冒頭一部だけを読んだ浅い要約
- 全セリフは保存したが、それを読んで理解する処理がない
- 24,803件のコメントも蓄積されているだけで、何も分析していない
- 結果として「動画を見たことがある人」には遠く及ばない理解度

## ゴール

「50本すべてのセリフ・映像・視聴者反応を深く読み込み、処理済みの知識としてDBに蓄積する」

この知識があることで、脚本制作時に「過去のバンカラジオをちゃんと理解した上での提案」ができるようになる。

---

## 処理の設計：3段階（ステップ1 → 2 → 3の順に積み上げ）

### ステップ1：1本ずつ深く読み解く（動画単位の深層分析）

50本それぞれに対して、以下の分析をGemini 2.5に実行させる。

**入力データ（1本あたり）：**
- 全セリフ（full_transcript）
- 既存VAS分析（brain_summary_v1 + brain_visual_audio_summary_v1）
- 視聴維持率データ（どこで視聴者が離脱したか）
- コメント（その動画についた全コメント）

**出力する知識（`brain_deep_analysis_v1`として保存）：**

```
1. scene_breakdown（シーン分解）
   - シーンごとの要約（何が起きたか）
   - 誰が何を言ったか（セリフの帰属）
   - そのシーンの役割（つかみ/展開/オチ/つなぎ）

2. character_behavior（キャラクター行動分析）
   - この動画での各キャラの役割
   - 口癖・決め台詞・リアクションパターン
   - キャラ同士の掛け合いの型（ボケツッコミ、権威逆転など）

3. comedy_mechanics（笑いの構造）
   - どこで笑いが起きているか（セリフ+タイミング）
   - 使われている笑いの技法（天丼、ツッコミ、予想裏切り、テンポ緩急）
   - エスカレーションの段階（どう盛り上がっていくか）

4. narrative_arc（物語の流れ）
   - 起承転結の各パートとその秒数
   - フック（最初の引き込み）が何をやっているか
   - 最大の山場はどこか
   - オチの型（どう終わるか）

5. audience_reception（視聴者の反応）
   - コメントから見える視聴者の感情・反応
   - 視聴維持率と内容の対応（どのシーンで離脱/継続が起きたか）
   - 視聴者がこの動画の「何」を評価しているか
```

**処理方法：**
- Gemini 2.5 Pro に全セリフ+VAS+コメント+維持率を渡し、上記5項目を一括分析
- 1本あたり1回のAPI呼び出し（全データを1つのプロンプトに収める）
- 結果は `assets.metadata_json` に `brain_deep_analysis_v1` キーで保存

**なぜこの順番か：**
- まず1本ずつ理解しないと、動画間の比較もパターン発見もできない
- 「この動画はこういう動画だ」を50本分持つことが次の土台になる

---

### ステップ2：動画間の比較とパターン発見（横断分析）

ステップ1で50本分の深層分析が揃ったら、動画同士を比較してパターンを抽出する。

**入力データ：**
- 50本分の `brain_deep_analysis_v1`
- 再生数・視聴維持率・コメント数などの成績データ

**出力する知識（`brain_cross_analysis_v1`として保存）：**

```
1. character_profiles（キャラクタープロファイル）
   - 全動画を通じた各キャラの特徴・口癖・行動パターン
   - どのキャラの組み合わせが強いか
   - キャラクター成長・変化の軌跡

2. format_patterns（フォーマットパターン）
   - 「もしも天才小学生が○○したら」の成功パターン
   - 「母×職業」の成功パターン
   - シリーズもの（逃走中、バンカレンジャー等）の発展パターン

3. comedy_patterns（笑いのパターン）
   - 再生数が伸びる笑いの型 vs 伸びない型
   - 視聴維持率が高い展開パターン
   - コメント数が多い（=反応が強い）仕掛け

4. structural_insights（構造的な気づき）
   - 最適な動画の長さ
   - フックの成功パターン
   - エスカレーションの理想的な段階数
   - オチの型と満足度の関係
```

**保存先：** 新テーブル `brain_knowledge` に保存（動画単位ではなくチャンネル全体の知識）

---

### ステップ3：知識の統合（バンカラジオとは何かの言語化）

ステップ1（個別理解）とステップ2（パターン発見）を統合して、「バンカラジオとはこういうチャンネルだ」という最終的な理解を作る。

**出力する知識（`brain_identity_v1`として保存）：**

```
1. バンカラジオのアイデンティティ
   - 何が他のチャンネルと違うのか
   - 視聴者が求めているもの
   - 絶対に外してはいけない要素

2. 脚本制作のためのルール集
   - こういう展開は伸びる
   - こういうキャラの使い方は強い
   - こういうフックの作り方がバンカラらしい
   - 避けるべきパターン
```

## 実装計画

### 今回実装する範囲：ステップ1のみ

まず1本を試作して確認してもらい、OKなら50本全部に展開する。

**新規ファイル：**
- `bankara_brain/analysis/deep_reading.py`
  - `build_deep_reading_prompt(asset, transcript, vas, comments, retention)` — プロンプト構築
  - `run_deep_reading(asset_id)` — 1本分の深層分析を実行
  - `run_deep_reading_all()` — 50本一括実行

**DBスキーマ変更：**
- なし（`assets.metadata_json` 内の `brain_deep_analysis_v1` キーに保存）

**CLIコマンド追加：**
- `deep-read --asset <ID>` — 1本だけ実行
- `deep-read --all` — 全件実行

### 実装ステップ

1. `deep_reading.py` を作成（プロンプト設計が最重要）
2. 1本の動画（コンビニ）で試作実行
3. 結果を確認してもらう
4. OKなら50本全部を実行（バックグラウンド）

### プロンプト設計のポイント

- 「要約しろ」ではなく「台本を読む脚本家として分析しろ」という指示
- セリフの帰属（誰のセリフか）の推定を明示的に求める
- 笑いのポイントを「なぜ面白いか」まで分析させる
- コメントと維持率は「答え合わせ」として使う（視聴者が実際にどう反応したか）
- バンカラジオ固有の文脈（キャラ設定、フォーマット）を前提知識として渡す

## リスクと対策

| リスク | 対策 |
|--------|------|
| Geminiの分析が浅い | プロンプトに具体例を含める。1本試作して深さを確認 |
| 自動字幕の精度問題 | 誤認識が多い箇所は「不明瞭」と明記させる |
| API料金 | Gemini 2.5 Flash を使用（Pro比で大幅に安い） |
| 50本の処理時間 | バックグラウンド実行。1本3-5分 × 50本 = 3-4時間 |
